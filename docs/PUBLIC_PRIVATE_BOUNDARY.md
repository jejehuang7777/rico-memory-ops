# Public vs Private Boundary

## Publish

- abstract schemas
- continuity rules
- handoff templates
- audit and demotion heuristics
- synthetic examples
- documentation that explains workflow intent

## Do not publish

- real user journals
- real task boards with personal details
- private family or house governance that depends on identity context
- any example that can reconstruct a person's life
- raw screenshots from private conversations
- direct exports of internal continuity files
- examples whose usefulness depends on knowing the real people involved

## Safe translation rule

Before publishing an example, ask:

1. Can another team use this without knowing our private story?
2. Does this example still work if all names and life details are removed?
3. Is the value in the method, not in the intimacy of the source material?

If the answer is no, keep it private.

## Recommended example style

- synthetic scenarios
- placeholder IDs
- neutral lane names
- short, reusable decision prompts
- transformed examples instead of direct lifts

## Release gate

Before anything leaves the private workspace, confirm all four:

1. No real names, no real personal history, no real diary text.
2. No screenshots or pasted excerpts from private continuity files.
3. The example is still useful after redaction and abstraction.
4. The method can stand on its own for an outside team.

## One-line rule

Publish the method.
Never publish the life behind it.
